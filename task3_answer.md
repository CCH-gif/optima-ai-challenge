Agent 开发中的真实挑战
在构建 Agent 时，我遇到过两个典型的“翻车”现场及解决方法：

规划失效（Tool Selection Error）：

痛点：模型在面临多个工具时，经常“胡乱投医”或者干脆复读机。

对策：我通过 LangGraph/LangChain 引入了更严谨的 ReAct 模式。核心在于精简工具描述（不要让 Prompt 太臃肿）并增加负面约束，明确告知模型哪些场景禁选 Web Search，必须走 RAG。

记忆缺失（Context Loss）：

痛点：多轮对话后，Agent 忘了最开始的租户报修地址。

对策：引入了 ConversationBufferWindowMemory。不仅是简单的对话堆叠，我还加入了一个 Summary 节点，每 5 轮对话自动总结一次关键信息，确保 Token 占用可控且核心上下文不丢。

### 详细步骤说明

| 步骤 | 负责人/系统 | 操作内容 |
|------|-------------|----------|
| 1. 邮件接收 | 邮件监听服务 | 通过 IMAP 定时拉取邮件 |
| 2. 内容解析 | NLP 模型 | 提取：姓名、房号、问题描述、附件图片 |
| 3. 意图识别 | 分类模型 | 判断是否为报修请求，提取关键信息（漏水位置、严重程度） |
| 4. 工单创建 | 工单系统 | 生成工单，关联租户信息、房屋信息 |
| 5. 优先级判定 | 规则引擎 | 根据关键词（"严重"、"漏水量大"）判定紧急程度 |
| 6. 自动回复 | 邮件服务 | 发送确认邮件，告知预计处理时间 |
| 7. 通知相关人 | 消息推送 | 通过微信/短信通知物业经理和维修班组 |
| 8. 派单分配 | 调度系统 | 根据维修班组位置、空闲情况自动派单 |
| 9. 维修处理 | 维修人员 | 现场维修，上传维修前后对比照片 |
| 10. 租户确认 | 消息推送 | 发送维修完成通知，请求确认 |
| 11. 满意度回访 | 问卷系统 | 自动发送满意度调查 |
| 12. 归档结算 | 工单系统 | 工单关闭，数据统计归档 |

### 关键技术点

- **邮件解析**：使用正则 + NLP 提取关键信息
- **图像识别**：如果附件包含漏水照片，用 CV 模型评估严重程度
- **定时任务**：每分钟拉取一次新邮件
- **状态机**：工单状态流转（待处理→处理中→已完成→已关闭）

## 3. Git rebase vs merge 区别

Merge（保全历史）：在 Feature Branch 合并回 Main 分支时使用。我需要那条清晰的“分叉再合并”的线，这代表了一个功能的生命周期，方便后续回滚整个功能块。

Rebase（线性美学）：

本地整理：在 Push 前，我会用 rebase -i 把那一堆“fix typo”、“test”之类的琐碎 Commit 压缩（Squash）成一个有意义的提交。

同步上游：当 Main 有更新时，我倾向于在 Feature 分支执行 git pull --rebase，避免产生无意义的 Merge Commit，保持历史干净。

### 黄金法则

> 永远不要对已推送到远程的 commits 进行 rebase

## 4. 接到陌生技术需求的第一步

**第一步：明确需求边界和背景**

具体行动：

1. **提问确认**：
   - 这个需求的业务目标是什么？
   - 谁是最终用户？
   - 有没有类似的现有功能可以参考？
   - 优先级和截止时间？

2. **技术调研**：
   - 查找相关文档、代码
   - 了解现有系统架构
   - 确定技术可行性

3. **评估复杂度**：
   - 拆解为子任务
   - 识别依赖项和风险点

**核心思路**：在动手之前，先确保理解"做什么"和"为什么做"，避免做无用功。

## 5. 对“没记录 = 没做”的深度理解
这句话在我的理解中，其实是关于“团队带宽”的优化。

减少重复劳动：今天我修了一个坑没记录，下周同事遇到同样的坑还得再踩一遍，这是对公司资源的极大浪费。

异步沟通的基础：好的文档和清晰的 Commit Message 能够让团队成员在不需要找我开会的情况下，就能理解我的代码逻辑。

自我保护：记录是复盘的依据。当项目出现问题时，清晰的轨迹能帮我们快速定位是逻辑问题还是环境问题，而不是互相“甩锅”。

## 6. 3小时完成任务后做什么

### 我的做法

1. **自我检查**
   - 是否有遗漏的边界情况？
   - 代码是否经过测试？
   - 是否有 lint/typecheck 问题？

2. **主动找事**
   - 回顾本周目标，看有没有可以提前完成的
   - 主动帮助同事 code review
   - 优化现有代码/文档
   - 学习新技术（但不影响正常工作）

3. **提前规划**
   - 规划明天的工作
   - 整理工作笔记

### 核心心态

不浪费时间，但也不"假装忙碌"。高效完成后，与其等任务不如主动找价值。
